你好，我是悦创。

如果你的 Agent 最近开始出现这些症状：

- 明明接了十几个工具，却越来越容易选错；
- 代码问题聊着聊着，混进了运维或文档上下文；
- 为了“更智能”加了多个 Agent，结果延迟和 Token 一起飙升；
- 出错后根本说不清，到底是路由、委派，还是上下文传递出了问题。

先别急着换更大的模型。

**很多时候，真正卡住系统的不是模型能力，而是架构没有跟着任务复杂度一起升级。**

但反过来也一样：多 Agent 不是默认答案。架构一旦选重了，你得到的可能不是更聪明的系统，而是更多模型调用、更长链路和更多失败点。

![从单体过载到模块化协作](https://raw.githubusercontent.com/AndersonHJB/PicGoImage/main/2026/0713/agent-architecture-guide/inline-chaos-to-order.png)

这篇文章不讲某一个框架的固定写法，只解决一个工程问题：

> **面对 Skills、Sub-Agent、Handoff 和 Router，究竟该怎么选？**

先给结论：

> 能用单 Agent + Tools 解决，就先别拆 Agent；工具和知识太多时先上 Skills；需要上下文隔离或并行专业分工时再用 Sub-Agent；输入天然跨多个领域时用 Router；流程必须按状态逐步推进时用 Handoff。

## 一、升级前先看两个核心信号

LangChain 的多 Agent 指南把核心问题归结为“上下文工程”：**每个处理单元应该看到什么，又不应该看到什么。**

当下面两个信号变得明显，多 Agent 才真正开始有价值。

### 信号 1：上下文已经开始互相污染

单个 Agent 同时承担代码、测试、安全、文档、运维等职责时，最常见的做法是继续往 System Prompt 里塞规则，再把所有工具一次性暴露给模型。

问题是：上下文窗口即使还没“满”，注意力也可能已经被稀释。

典型表现包括：

- 工具选择错误率上升；
- 回答开始“串味”；
- 不相关历史进入当前推理；
- Prompt 改动牵一发而动全身。

这时你需要的不是更多文字，而是**策略性地分发上下文**。

### 信号 2：能力需要被不同团队独立维护

安全团队希望独立迭代审计能力，运维团队要维护日志与监控工具，研发团队又在持续更新代码检索规则。

如果所有能力都绑在同一个 Agent 里，每次升级都要一起测试、一起发布，组织边界会逐渐变成系统耦合。

当不同团队需要独立拥有 Prompt、工具和评测集时，拆分 Agent 才能把责任边界真正落到工程结构里。

此外，还有一个常见的加速信号：**任务能否被拆成相互独立、可以并行完成的子问题。** 如果答案是否定的，多 Agent 通常也很难带来明显收益。

## 二、四种模式，一眼分清

![四种 Agent 架构模式对比](https://raw.githubusercontent.com/AndersonHJB/PicGoImage/main/2026/0713/agent-architecture-guide/architecture-map.png)

### 1. Skills：先给单 Agent 一个“按需加载”的能力柜

Skills 仍然是单 Agent。区别在于，Agent 启动时只知道技能名称和用途，真正需要时才加载对应的知识、步骤和约束。

它解决的是：**能力很多，但一次任务只会用到少数几种。**

例如，一个编码助手拥有部署、代码审查、数据库迁移、测试和文档等十几项能力。与其把全部说明写进 System Prompt，不如让它根据任务按需加载 `SKILL.md`。

适合：

- 工具多、知识多，但任务仍属于同一主领域；
- 对话连续性很重要；
- 希望以最低复杂度实现渐进式能力扩展。

代价：所有技能最终仍共享同一个对话上下文。连续加载多个大技能后，Token 还是会累积，隔离能力也有限。

一句话记忆：

> **Skill 是给同一个人按需发操作手册。**

### 2. Sub-Agent：让主 Agent 把专业任务委派出去

Sub-Agent 模式通常由一个 Supervisor 负责理解目标、拆分任务、委派专家并汇总结果。每个 Sub-Agent 拥有独立上下文，只接收完成当前子任务所需的信息。

它真正的价值首先是**上下文隔离**，其次才是并行。

适合：

- 任务可以拆成多个相对独立的专业子任务；
- 每个领域需要大量专属资料或工具；
- 主 Agent 只需要接收压缩后的结论；
- 多个子任务可以并行执行。

代价：多一层委派与汇总，就多一层调用、延迟和失败点。Supervisor 如果不会拆任务，再强的 Sub-Agent 也只会高效地跑偏。

一句话记忆：

> **Sub-Agent 是老板把研究、分析、写作分别交给不同专家，再统一验收。**

Anthropic 的 Research 系统就是典型案例：Lead Agent 制定研究策略，多个 Sub-Agent 并行搜索，最后再聚合结果和引用。它适合“广度优先”、可以沿多条独立路径同时探索的任务。

### 3. Router：先判断“这是什么问题”，再决定交给谁

Router 的职责不是完成任务，而是分类、拆分和分发。

例如用户问：“公司的退货政策是什么？最近销售数据怎么样？”Router 可以把政策问题交给文档 Agent，把销售问题交给数据 Agent，再把两部分结果合成一个回答。

适合：

- 一个请求天然跨多个知识域或数据源；
- 子问题之间依赖很少，可以并行；
- 专家边界清晰，路由规则可评测。

代价：Router 通常是无状态的，每轮都要重新判断；一旦分类错了，后面的专家越强，结果可能离目标越远。

一句话记忆：

> **Router 是总机：先分流，再并行，最后合并。**

### 4. Handoff：让控制权随着状态逐步交接

Handoff 的核心不是并行，而是状态切换。

客服工单就是最直观的例子：前台 Agent 负责收集信息；条件满足后交给技术支持；仍未解决时再交给高级工程师。每一阶段拥有不同 Prompt、工具和退出条件。

适合：

- 流程有明确阶段；
- 后续能力必须在满足前置条件后才能解锁；
- 用户需要保持连续、自然的多轮对话体验。

代价：它天然偏顺序执行，不适合需要广泛并行检索的任务。工程上最容易出问题的地方，是阶段没有清晰的完成条件，导致流程卡住或错误跳转。

一句话记忆：

> **Handoff 是接力赛：当前角色完成自己的阶段，再把状态和控制权交给下一棒。**

## 三、别只看概念：三组数据告诉你真实代价

LangChain 在官方多 Agent 文档中，对四种模式做了三类对比。这里最值得记住的不是某一个绝对数字，而是不同结构的成本趋势。

![四种模式的调用与 Token 对比](https://raw.githubusercontent.com/AndersonHJB/PicGoImage/main/2026/0713/agent-architecture-guide/benchmark-comparison.png)

### 场景 1：一次简单请求

完成一个单一任务时，Sub-Agent 需要 4 次模型调用；Skills、Handoff、Router 都是 3 次。多出来的一次，来自结果回到主 Agent 再统一输出。

这说明：**简单任务里，中心化编排通常只有开销，没有收益。**

### 场景 2：同类请求连续出现

第二轮相同请求中，Skills 和 Handoff 可以复用已经加载的技能或当前状态，两轮合计 5 次调用；Router 为 6 次；Sub-Agent 因为子代理通常从新上下文启动，两轮共 8 次。

这说明：**连续对话里，有状态模式更省调用。**

### 场景 3：跨多个专业领域

当任务要求对比 Python、JavaScript 和 Rust，并假设每个领域有约 2,000 Token 的资料时：

- Sub-Agent：约 9K Token；
- Router：约 9K Token；
- Handoff：约 14K+ Token；
- Skills：约 15K Token。

原因很直观：Sub-Agent 和 Router 能让每个专家只看到与自己相关的上下文；Skills 虽然模型调用少，但加载后的多份资料会在后续调用中反复进入上下文。

这说明：**多领域任务里，上下文隔离往往比少一次模型调用更值钱。**

## 四、90.2% 的提升，为什么不能被误读

Anthropic 在 2025 年发布的多 Agent Research 工程复盘中提到：在其内部研究评测上，以 Claude Opus 4 作为 Lead Agent、Claude Sonnet 4 作为 Sub-Agent 的系统，比单 Agent Claude Opus 4 高出 90.2%。

这个数字很亮眼，但必须同时读完后半句：

- 这是**内部研究评测**，不是所有 Agent 任务的通用结论；
- Agent 通常消耗普通聊天约 4 倍 Token；
- 多 Agent 系统通常消耗普通聊天约 15 倍 Token；
- 在 BrowseComp 分析中，Token 用量、工具调用次数和模型选择解释了 95% 的性能差异，其中 Token 用量单独解释了约 80%。

换句话说，多 Agent 的能力提升并不是“免费协作”，而是在用更多独立上下文、更高并行度和更多模型调用换取更大的搜索覆盖面。

它尤其适合：高价值、重检索、可并行、信息量超过单上下文承载能力的任务。

它不适合：所有 Agent 必须共享完整上下文、任务强依赖且高度顺序化、子任务很难独立验收的场景。Anthropic 也明确指出，大多数编码任务的可并行部分通常少于研究任务。

所以，90.2% 不是“赶快上多 Agent”的广告，而是一条更严格的工程提醒：

> **只有当任务价值足以覆盖约 15 倍 Token 成本，并且问题真的能被并行拆解时，这笔账才可能成立。**

## 五、从单 Agent 到混合架构：正确的升级顺序

![Agent 架构升级决策树](https://raw.githubusercontent.com/AndersonHJB/PicGoImage/main/2026/0713/agent-architecture-guide/decision-tree.png)

一个更稳妥的演进路径通常是：

### 第一步：单 Agent + Tools

先验证需求和任务闭环。工具少、上下文短、职责单一时，不需要额外设计模式。

### 第二步：单 Agent + Skills

当工具与说明越来越多，但任务仍在同一领域，先用 Skills 做渐进式加载。它通常是成本最低、风险最小的一次升级。

### 第三步：引入 Sub-Agent 或 Router

如果问题已经变成多个专业领域的组合：

- 需要主 Agent 持续拆解、追问和验收，用 Sub-Agent；
- 输入可以先分类或拆成独立查询，用 Router。

### 第四步：为顺序流程加入 Handoff

当不同阶段拥有不同权限、工具和完成条件时，再把 Handoff 引入流程。

### 第五步：只在必要时组合模式

成熟系统往往不是“四选一”，而是组合使用：Router 负责分类，Supervisor 调度 Sub-Agent，某些业务流程再通过 Handoff 推进；每个 Agent 内部还可以继续用 Skills 按需加载知识。

但组合架构应该是需求长出来的结果，不是项目第一天画出来的愿望清单。

## 六、真正落地时，守住这 6 条工程规则

### 1. 先找错误类型，再选架构

“工具太多”不等于必须多 Agent。先区分问题来自工具选择、上下文污染、流程状态，还是跨域并行。不同错误对应不同结构。

### 2. 给每个边界写清输入和输出

Sub-Agent 接收什么？必须返回什么？Router 的分类标签有哪些？Handoff 的触发条件是什么？边界如果不能被结构化描述，就很难被测试。

### 3. 在每次交接前设置检查点

不要让错误结果直接进入下一阶段。至少检查完整性、事实依据、格式和失败状态，再允许路由、汇总或交接。

### 4. 记录完整调用链

多 Agent 的失败往往不是一个错误，而是一串小偏差。日志里应该能看到每个 Agent 的输入、工具调用、关键输出、耗时和 Token，而不只是最终答案。

### 5. 按任务价值分配“努力程度”

简单问题只调用一个 Agent；中等任务再并行少量专家；复杂研究才扩大搜索宽度。不要让所有请求默认走最昂贵的路径。

### 6. 把 Token 当作架构指标

模型调用数、总 Token、P95 延迟、路由正确率、交接失败率，都应该进入评测。没有这些数据，多 Agent 很容易变成“看起来更专业”的昂贵黑盒。

## 七、最后做一个真实选择题

假设你正在搭建企业内部技术助手，它已经接入 15+ 个工具，能查代码、文档、日志和监控。用户开始反馈回答跑偏、上下文串味，调试一次问题还要反复重放对话。

下一步应该直接上 Supervisor + Sub-Agent 吗？

我的答案是：**先看错误分布。**

- 如果问题主要来自工具和说明太多，但多数请求仍是单领域，先用 Skills；
- 如果代码、文档、运维的上下文确实互相污染，再拆 Sub-Agent；
- 如果用户经常一次提出跨领域问题，并且各子问题可并行，加入 Router；
- 如果排障必须经过“收集信息 → 初步诊断 → 高级处理”，再用 Handoff 管理状态。

架构升级的触发信号，不应该只是“工具数量到了 15 个”，而应该是：**错误类型已经稳定地指向某种结构性瓶颈。**

## 写在最后

从 Sub-Agent 到 Multi-Agent，不是一条“越复杂越高级”的升级路线，而是一组用来解决不同工程问题的结构工具。

你真正需要记住的只有五句话：

1. 能单 Agent 解决，就别急着拆；
2. 能用 Tools 解决，就先别加 Agent；
3. 能用 Skills 延迟加载，就别把所有知识塞进 Prompt；
4. 多 Agent 的第一价值是上下文隔离，第二才是并行；
5. 最好的架构，不是最复杂的架构，而是恰好消化当前复杂度的架构。

2026 年，Agent 的竞争已经从“会不会写 Prompt”，进入“能不能把复杂度组织好”的阶段。

模型决定能力上限，结构决定你能不能稳定地接近它。

你现在的项目，最接近哪一种瓶颈：上下文污染、工具过多、跨域并行，还是状态流转？欢迎把你的场景写在评论区。
